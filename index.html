O Sistema ViajeBem

O que é o Projeto

Uma plataforma moderna e intuitiva de agendamento de viagens e gestão centralizada de reservas, construída com tecnologias de ponta.

O Problema

Elimina a fricção e a complexidade manual no processo de reserva, permitindo que o usuário encontre viagens com rapidez e segurança.

Recursos Principais
Gestão de Vagas em Tempo Real Cálculo dinâmico de disponibilidade para evitar overbooking.
Consistência de Regras Automação de políticas de cancelamento e prazos de partida.
Controle de Acesso (RBAC) Distinção clara entre Cliente (busca e reserva) e Administrador/Líder (gestão de frota e usuários).

Evolução e Escolhas Tecnológicas

Stack Identificada
Next.js 15.1.3 (App Router) React 19.0.0
Tailwind CSS v3.4.17 Estilização Base
Vitest + Testing Library Ambiente de Testes
Decisões de Engenharia
Preservação do Tailwind v3

Optamos por manter a versão 3 estável para garantir a integridade do design system já construído, evitando as quebras de retrocompatibilidade da v4.

Paradigma Open Code Shadcn/ui

Em vez da CLI automatizada do Shadcn, adotamos a filosofia pura do ecossistema. Injetamos o código-fonte dos primitivos (Button e Input) diretamente em src/components/ui/.

Independência Arquitetural

Controle total sobre o código visual — eliminando caixas-pretas no node_modules e mantendo a interface flexível e compatível com nossos testes comportamentais.

Camada de Domínio: O Coração do Negócio

O que é

O núcleo estável e central do sistema, puramente focado em regras de negócio, sem qualquer dependência de frameworks, bibliotecas de UI ou rede.

O que está incluso

Entidades ricas (Viagem, Reserva) com comportamentos calculados; Value Objects (Email, StatusReserva); e Erros Customizados para falhas semânticas.

Onde está localizado

Localizado em frontend/src/domain/. Arquivos organizados nas pastas entities/, value-objects/ e errors/.

De que forma / Justificativa

Implementado com TypeScript Puro e Acoplamento Zero. Garante que regras vitais de negócio sejam imunes a mudanças em ferramentas externas.

Evidência no Código Fonte

Camada de Aplicação: Intenções e Orquestração

O que é

A camada responsável por implementar os fluxos de software que satisfazem as intenções do usuário (Casos de Uso), orquestrando as entidades do domínio.

O que está incluso

Classes de Use Cases como LoginUseCase e CriarReservaUseCase, que coordenam a lógica entre a entrada de dados, regras de negócio e persistência.

Onde está localizado

Localizado em frontend/src/application/use-cases/.

De que forma / Justificativa

Implementado via Inversão de Dependência (SOLID). Os casos de uso são "tecnologicamente cegos", recebendo apenas interfaces no construtor. Não sabem detalhes de rede ou persistência externa, facilitando testes e portabilidade.

  async execute(credenciais: LoginDTO): Promise<AuthResponseDTO> {
    try {
      const response = await this.authRepository.login(credenciais)
      this.tokenStorage.save(response.accessToken, {
        id: response.usuario.id,
        nome: response.usuario.nome,
        apelido: response.usuario.apelido,

        tipo: response.usuario.tipo,

Camada de Infraestrutura: Tecnologia e Integração

O que é

A borda tecnológica que lida com o mundo externo (APIs, Storage de cookies, etc.), concretizando os contratos do domínio.

O que está incluso

HttpClient (Fetch Nativo), repositórios concretos (ex: AuthHttpRepository) e o container central de injeção de dependências.

Onde está localizado

Localizado em frontend/src/infrastructure/, contendo pastas como http/, repositories/ e di/.

De que forma / Justificativa

Design Sustentável: Usamos Fetch API nativa eliminando dependência do Axios. A fiação centralizada no container.ts cria um local único para montagem das dependências.

Testes de Interface e Qualidade

O que é

A garantia automatizada de que os componentes da interface visual e as regras de negócio subjacentes funcionam em perfeita harmonia.

O que está incluso

Testes de unidade da camada de domínio e testes de integração de UI simulando fluxos completos com a React Testing Library.

Onde está localizado

Distribuidos nas pastas __tests__/ do projeto, especificamente em app/login/__tests__/page.test.tsx.

De que forma / Justificativa

Simulação Real do Usuário: Usando Vitest e ambiente JSDOM para renderização em memória. Garante que a integração entre componentes de UI, Custom Hooks e Use Cases esteja 100% blindada.
